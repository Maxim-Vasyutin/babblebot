# Queue Cat — Техническая спецификация

> Версия: 1.0 | Дата: 2026-07-08 | Статус: Production-ready
> Проект: Telegram-бот заказов для очередей на АЗС (Bubble Cat, Пенза)

---

## 0. Обзор проекта

### Что это
Telegram-бот, принимающий заказы напитков и сэндвичей от водителей, стоящих в
очередях на АЗС. Клиент собирает корзину, указывает машину и ориентир в очереди,
заказ прилетает карточкой в рабочий Telegram-чат заведения, курьер доставляет к
машине. Оплата при получении. Дополнительно бот собирает статистику спроса на
позиции, которых сегодня нет в наличии.

### Стек
- **Runtime:** Next.js 16 (App Router), TypeScript — единственный API route
  работает как Telegram webhook на Vercel (serverless function)
- **Backend:** Supabase (PostgreSQL 15). Доступ к БД — только server-side, через
  service-role ключ. Публичного браузерного клиента нет → RLS не является контуром
  безопасности (тем не менее RLS включается с deny-all для defense-in-depth)
- **Деплой:** Vercel (webhook endpoint)
- **Интеграции:** Telegram Bot API (getUpdates отключён, работаем на webhook;
  sendMessage / editMessageText / answerCallbackQuery)
- **Валидация:** Zod
- **HTTP к Telegram:** нативный fetch (Node 20+ на Vercel)

### Модель работы (важно для архитектуры)
Telegram webhook доставляет каждое обновление (update) отдельным POST-запросом на
`/api/telegram`. Функция обрабатывает ОДИН update и завершается. Состояния между
запросами нет в памяти — **всё состояние диалога хранится в БД** (таблица
`user_sessions`, поле `state` + `context` JSONB). Это ключевое следствие
serverless: FSM пользователя персистентна в Postgres, а не в RAM.

### Роли
| Роль | Описание | Как определяется | Доступ |
|------|----------|------------------|--------|
| client | Водитель в очереди | Любой, кто написал боту в личку | Меню, корзина, оформление, «О нас» |
| admin | Сотрудник заведения / курьер | Сообщение пришло из чата с `chat_id` == сохранённый admin_chat_id | Кнопки статуса заказов, /стоп, /стат |

Разделение по источнику апдейта: приватный чат (`chat.type == "private"`) → клиент;
сообщение/‑callback из группы с известным `admin_chat_id` → админ. Никакой
парольной авторизации нет — админом становится тот, кто физически добавлен в
рабочий чат и выполнил `/init`.

### Маршруты (endpoints приложения)
| Путь | Метод | Назначение | Доступ |
|------|-------|-----------|--------|
| `/api/telegram` | POST | Приём webhook-обновлений от Telegram | Telegram (проверка secret_token) |
| `/api/health` | GET | Проверка живости для мониторинга | Публичный |

Единственная «страница» — вебхук. UI приложения = диалог бота (Блок 4).

## БЛОК 1: User Stories

### US-001: Первый вход по QR
**Как** водитель, скучающий в очереди на АЗС,
**я хочу** отсканировать QR с флаера и сразу увидеть, что можно заказать,
**чтобы** понять, стоит ли оно того, за 5 секунд без регистрации.

**Сценарий:**
1. Водитель наводит камеру на QR флаера → открывается `t.me/<bot>` → жмёт «Запустить»
2. Бот шлёт приветствие + фото + 3 кнопки: «🧋 Меню», «🛒 Мой заказ», «ℹ️ О нас»
3. Водитель жмёт «🧋 Меню»
4. Бот показывает 6 позиций «в наличии» с ценами и секцию «сегодня нет» с 👀
5. При первом апдейте создаётся строка в `user_sessions` (state = 'browsing')

**Критерии приёмки:**
- [ ] `/start` работает без каких-либо предварительных данных о пользователе
- [ ] Приветствие приходит < 2 сек после запуска
- [ ] Ни на одном шаге до оформления бот НЕ просит регистрацию/телефон
- [ ] Если пользователь уже писал раньше — приветствие то же, старая корзина очищена

### US-002: Сбор заказа с вариантом позиции
**Как** водитель, выбравший айс-латте,
**я хочу** указать, с сиропом или без, и сколько штук,
**чтобы** получить именно то, что хочу, и курьер не переспрашивал.

**Сценарий:**
1. В меню водитель тапает «Айс-латте — 250₽»
2. Бот отвечает выбором варианта: кнопки «Без сиропа» / «С сиропом»
3. Водитель жмёт «С сиропом»
4. Позиция добавляется в корзину (кол-во 1), бот показывает корзину с [−]/[+]
5. Водитель жмёт [+] → количество 2, сумма пересчитана
6. Позиции без вариантов (лимонад, тоник, сэндвичи) добавляются сразу, без шага 2

**Критерии приёмки:**
- [ ] Латте требует выбора сироп/без; чай требует зелёный/чёрный — до попадания в корзину
- [ ] Сироп не меняет цену (обе кнопки → 250 ₽)
- [ ] Один и тот же напиток с разными вариантами = две отдельные строки корзины
- [ ] Количество на позицию: 1–20; при попытке [−] ниже 1 — позиция удаляется из корзины
- [ ] Сумма корзины пересчитывается при каждом изменении

### US-003: Оформление заказа
**Как** водитель, собравший корзину,
**я хочу** быстро указать, где я стою и какая у меня машина,
**чтобы** курьер меня нашёл, не блуждая по очереди.

**Сценарий:**
1. В корзине водитель жмёт «✅ Оформить»
2. Бот: «Где вы стоите?» + кнопки: «У въезда», «Середина», «Хвост», «Напишу словами»
3. Водитель жмёт «Середина» (или «Напишу словами» → вводит текст ориентира)
4. Бот: «Машина — одним сообщением: цвет, марка, номер»
5. Водитель пишет «белая гранта а123вс»
6. Бот: «Как оплатите?» + кнопки «СБП», «Наличные», «Карта (терминал)»
7. Водитель жмёт «Наличные»
8. Бот показывает итоговое подтверждение с номером заказа
9. Карточка заказа улетает в admin-чат

**Критерии приёмки:**
- [ ] Ориентир — обязателен (кнопка или текст); пустой словесный ориентир → повтор вопроса
- [ ] Машина — обязательный непустой текст 3–100 символов
- [ ] Оплата — обязательный выбор из трёх кнопок
- [ ] Клиент получает номер заказа (сквозной инкремент) и состав
- [ ] Если admin_chat не привязан (`/init` не выполнен) — заказ всё равно сохраняется,
      клиент получает подтверждение, ошибка логируется (заказ не теряется)

### US-004: Повторный заказ с той же машиной
**Как** водитель, заказывающий второй раз,
**я хочу** не вводить машину заново,
**чтобы** оформить за 20 секунд.

**Сценарий:**
1. Водитель, уже делавший заказ, собирает новую корзину, жмёт «Оформить»
2. После ориентира бот: «Машина та же — белая Гранта А123ВС?» + [Да] / [Другая]
3. «Да» → переход сразу к оплате; «Другая» → обычный ввод машины
4. Последняя использованная машина хранится в `user_sessions.last_car`

**Критерии приёмки:**
- [ ] Предложение прошлой машины появляется только если `last_car` не пуст
- [ ] «Да» пропускает шаг ввода машины
- [ ] Новая машина после «Другая» перезаписывает `last_car`

### US-005: Приём и ведение заказа курьером
**Как** курьер в рабочем чате,
**я хочу** видеть заказы карточками и отмечать статус в один тап,
**чтобы** не путаться, какие заказы собраны и доставлены.

**Сценарий:**
1. В admin-чат приходит карточка: «🆕 Заказ #7», состав, сумма, машина, ориентир, оплата
2. Под карточкой кнопки: «🚚 Взял», «✅ Доставлен», «❌ Отмена»
3. Курьер жмёт «🚚 Взял» → карточка редактируется: статус меняется на «🚚 В работе», кто взял
4. Жмёт «✅ Доставлен» → статус «✅ Доставлен», время; кнопки убираются
5. Статус пишется в `orders.status`

**Критерии приёмки:**
- [ ] Нажатие кнопки редактирует ту же карточку (editMessageText), а не шлёт новую
- [ ] Статусы: new → in_progress → delivered; либо любой → cancelled
- [ ] После delivered/cancelled кнопки статуса пропадают
- [ ] Виден Telegram-username того, кто менял статус (в тексте карточки)

### US-006: Управление наличием позиции
**Как** сотрудник заведения,
**я хочу** быстро снять кончившуюся позицию с продажи,
**чтобы** бот не принимал заказы на то, чего нет (по 5 шт/позиция — кончается быстро).

**Сценарий:**
1. Сотрудник в admin-чате пишет `/стоп`
2. Бот отвечает клавиатурой из всех полевых позиций с их статусом (✅/⛔)
3. Жмёт «Айс-латте ✅» → позиция становится недоступной, кнопка меняется на «Айс-латте ⛔»
4. У клиентов латте уезжает из «в наличии» в «сегодня нет 👀»
5. Повторный тап возвращает позицию в наличие

**Критерии приёмки:**
- [ ] `/стоп` работает только из admin-чата; в личке боту — игнор/подсказка
- [ ] Снятая позиция не добавляется в корзину; при попытке — «Уже разобрали 🙈»
- [ ] Позиция в «сегодня нет» кликабельна как «хочу 👀» и растит счётчик спроса
- [ ] Состояние наличия переживает перезапуск (хранится в `menu_items.available`)

### US-007: Учёт спроса на отсутствующие позиции
**Как** владелец,
**я хочу** видеть, что чаще всего хотят из того, чего сегодня нет,
**чтобы** решить, что докупить и привезти завтра.

**Сценарий:**
1. Клиент в секции «сегодня нет» тапает «Ядерный зелёный 👀»
2. Бот: «Записали! 🐾 Наберётся спрос — привезём» + ссылка на группу
3. `demand_clicks` для позиции +1
4. Сотрудник пишет `/стат` → среди прочего топ позиций по «хочу»

**Критерии приёмки:**
- [ ] Каждый тап 👀 = +1 (без дедупликации по пользователю в MVP — считаем интерес)
- [ ] Позиции «сегодня нет» владелец дозаполняет сам (в MVP список может быть пуст/мал)
- [ ] `/стат` показывает топ по убыванию

### US-008: Дневная сводка
**Как** владелец в конце смены,
**я хочу** одной командой увидеть итоги дня,
**чтобы** понять выручку и спрос без ручного подсчёта.

**Сценарий:**
1. В admin-чате `/стат`
2. Бот отвечает: число заказов за сегодня, сумма, разбивка по способам оплаты,
   топ заказанных позиций, топ «хочу 👀»
3. Расчёт по заказам со статусом != cancelled за текущую дату (TZ Europe/Moscow)

**Критерии приёмки:**
- [ ] Считаются заказы с created_at за сегодня по Москве
- [ ] Отменённые (cancelled) в выручку не входят
- [ ] Если заказов нет — «Сегодня пока пусто»

## БЛОК 2: Data Model

### Диаграмма связей
```
menu_items      1──N order_items (menu_item_id, SET NULL — позиция может быть удалена)
orders          1──N order_items (order_id, CASCADE)
user_sessions   — по telegram user_id, хранит FSM-состояние и корзину
app_settings    — key/value, singleton-настройки (admin_chat_id, order_counter)
demand_clicks   — счётчик «хочу» по позициям, которых нет
```
Все денежные значения — INTEGER в копейках. 250 ₽ = 25000.

### Замечание по безопасности
Бот ходит в БД только с сервера, service-role ключом. Публичного anon-клиента нет,
поэтому RLS не является защитным контуром. Тем не менее для каждой таблицы включаем
RLS и НЕ создаём разрешающих политик для anon/authenticated — это deny-all для
любого случайного публичного доступа (defense-in-depth). Service-role обходит RLS
штатно.

### Таблица: app_settings
```sql
-- Singleton key/value настройки приложения.
-- Ключи: 'admin_chat_id' (BIGINT как текст), 'order_counter' (сквозной номер заказа).
CREATE TABLE app_settings (
  key TEXT PRIMARY KEY,
  value TEXT NOT NULL,
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

ALTER TABLE app_settings ENABLE ROW LEVEL SECURITY;
-- deny-all: политик для anon/authenticated не создаём. Доступ только service-role.

-- Сквозной счётчик заказов реализуем атомарно через UPDATE ... RETURNING (см. Блок 3),
-- начальное значение:
INSERT INTO app_settings (key, value) VALUES ('order_counter', '0')
  ON CONFLICT (key) DO NOTHING;
```

### Таблица: menu_items
```sql
-- Каталог позиций. Полевые 6 позиций сидируются как available=true.
-- Позиции "сегодня нет" (для статистики спроса) владелец добавляет позже как
-- available=false, field_item=false.
CREATE TABLE menu_items (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  slug TEXT NOT NULL UNIQUE,          -- стабильный ключ для callback_data, напр. 'ice_latte'
  title TEXT NOT NULL,               -- 'Айс-латте'
  price_kop INTEGER NOT NULL DEFAULT 0 CHECK (price_kop >= 0),
  variant_group TEXT CHECK (variant_group IN ('syrup', 'tea')), -- NULL = без вариантов
  field_item BOOLEAN NOT NULL DEFAULT false, -- true = входит в полевое меню дня
  available BOOLEAN NOT NULL DEFAULT false,  -- true = сейчас в наличии
  sort_order INTEGER NOT NULL DEFAULT 100,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_menu_items_available ON menu_items(available) WHERE available = true;
CREATE INDEX idx_menu_items_field ON menu_items(field_item);
CREATE INDEX idx_menu_items_sort ON menu_items(sort_order);

ALTER TABLE menu_items ENABLE ROW LEVEL SECURITY;

-- Триггер updated_at
CREATE EXTENSION IF NOT EXISTS moddatetime SCHEMA extensions;
CREATE TRIGGER menu_items_moddatetime
  BEFORE UPDATE ON menu_items
  FOR EACH ROW EXECUTE FUNCTION extensions.moddatetime(updated_at);
```

### Таблица: user_sessions
```sql
-- FSM-состояние диалога + корзина. Ключ — Telegram user_id (BIGINT).
-- Serverless: между запросами нет памяти, поэтому весь диалог живёт здесь.
CREATE TABLE user_sessions (
  user_id BIGINT PRIMARY KEY,          -- Telegram from.id
  username TEXT,                        -- @username на момент последнего апдейта (может быть NULL)
  state TEXT NOT NULL DEFAULT 'browsing'
    CHECK (state IN ('browsing','choosing_variant','cart',
                     'checkout_landmark','checkout_landmark_text',
                     'checkout_car','checkout_payment')),
  context JSONB NOT NULL DEFAULT '{}'::jsonb, -- временные данные шага (напр. выбранный slug, ориентир)
  cart JSONB NOT NULL DEFAULT '[]'::jsonb,    -- массив строк корзины (см. формат ниже)
  last_car TEXT,                        -- последняя машина для быстрого повтора
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_user_sessions_updated ON user_sessions(updated_at);

ALTER TABLE user_sessions ENABLE ROW LEVEL SECURITY;
-- deny-all.

CREATE TRIGGER user_sessions_moddatetime
  BEFORE UPDATE ON user_sessions
  FOR EACH ROW EXECUTE FUNCTION extensions.moddatetime(updated_at);
```

**Формат строки корзины (элемент массива `cart`):**
```json
{
  "slug": "ice_latte",
  "title": "Айс-латте",
  "variant": "С сиропом",
  "price_kop": 25000,
  "qty": 2
}
```
Строки с одинаковым `slug` + `variant` объединяются (растёт `qty`). Разный вариант —
разные строки.

### Таблица: orders
```sql
CREATE TABLE orders (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  order_number INTEGER NOT NULL,        -- сквозной человекочитаемый номер (#7)
  user_id BIGINT NOT NULL,              -- Telegram from.id (не FK на auth.users — юзеров Supabase нет)
  username TEXT,
  items JSONB NOT NULL,                 -- снапшот корзины на момент заказа (формат строки корзины)
  total_kop INTEGER NOT NULL CHECK (total_kop >= 0),
  landmark TEXT NOT NULL,               -- 'Середина очереди' или произвольный текст
  car TEXT NOT NULL,                    -- 'белая гранта а123вс'
  payment_method TEXT NOT NULL CHECK (payment_method IN ('sbp','cash','terminal')),
  status TEXT NOT NULL DEFAULT 'new'
    CHECK (status IN ('new','in_progress','delivered','cancelled')),
  admin_message_id BIGINT,              -- id карточки в admin-чате (для editMessageText)
  handled_by TEXT,                      -- @username курьера, менявшего статус
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_orders_created ON orders(created_at);
CREATE INDEX idx_orders_status ON orders(status);
CREATE INDEX idx_orders_number ON orders(order_number);

ALTER TABLE orders ENABLE ROW LEVEL SECURITY;
-- deny-all.

CREATE TRIGGER orders_moddatetime
  BEFORE UPDATE ON orders
  FOR EACH ROW EXECUTE FUNCTION extensions.moddatetime(updated_at);
```
Комментарий: `items` — снапшот (JSONB), а не FK-связь, намеренно. Заказ должен
навсегда сохранить, что именно и по какой цене куплено, даже если позицию в
`menu_items` потом переименуют/удалят. `order_items` как отдельную таблицу в MVP
НЕ вводим — снапшот проще и достаточен для статистики через JSONB-функции.

### Таблица: demand_clicks
```sql
-- Счётчик «хочу 👀» по позициям, которых нет в наличии.
-- Отдельная таблица (а не поле в menu_items), чтобы считать спрос и по позициям,
-- которых ещё нет в каталоге (владелец может завести их позже; в MVP пишем по slug).
CREATE TABLE demand_clicks (
  slug TEXT PRIMARY KEY,       -- ключ позиции (может ссылаться на menu_items.slug или быть новым)
  title TEXT NOT NULL,         -- человекочитаемое имя для /стат
  clicks INTEGER NOT NULL DEFAULT 0 CHECK (clicks >= 0),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

ALTER TABLE demand_clicks ENABLE ROW LEVEL SECURITY;
-- deny-all.

CREATE TRIGGER demand_clicks_moddatetime
  BEFORE UPDATE ON demand_clicks
  FOR EACH ROW EXECUTE FUNCTION extensions.moddatetime(updated_at);
```

### RPC: атомарный сквозной номер заказа
```sql
-- Гонки между параллельными serverless-вызовами закрываем атомарным UPDATE.
CREATE OR REPLACE FUNCTION next_order_number()
RETURNS INTEGER AS $$
DECLARE
  n INTEGER;
BEGIN
  UPDATE app_settings
    SET value = (value::int + 1)::text, updated_at = NOW()
    WHERE key = 'order_counter'
    RETURNING value::int INTO n;
  RETURN n;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;
```

### RPC: атомарный инкремент спроса
```sql
CREATE OR REPLACE FUNCTION bump_demand(p_slug TEXT, p_title TEXT)
RETURNS VOID AS $$
BEGIN
  INSERT INTO demand_clicks (slug, title, clicks)
    VALUES (p_slug, p_title, 1)
    ON CONFLICT (slug)
    DO UPDATE SET clicks = demand_clicks.clicks + 1, updated_at = NOW();
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;
```

## БЛОК 3: Команды и Callback-протокол бота

> Аналог REST API для веб-приложения. Telegram-бот «общается» через входящие
> updates и исходящие вызовы Bot API. Единственный HTTP-endpoint приложения —
> вебхук; ниже описан протокол его входа и внутренняя маршрутизация.

### Endpoint: `POST /api/telegram`
**Описание:** принимает Update от Telegram, маршрутизирует по типу.
**Авторизация:** заголовок `X-Telegram-Bot-Api-Secret-Token` должен равняться
`process.env.TELEGRAM_WEBHOOK_SECRET`. Иначе — 401.

**Пример входа (нажатие inline-кнопки):**
```json
{
  "update_id": 123456789,
  "callback_query": {
    "id": "cb_abc",
    "from": { "id": 555000111, "username": "driver_ivan" },
    "message": { "message_id": 42, "chat": { "id": 555000111, "type": "private" } },
    "data": "add:ice_latte"
  }
}
```
**Ответ (всегда):** `200 OK` с телом `{ "ok": true }`. Telegram не должен получать
не-2xx (иначе ретраит) — любые внутренние ошибки логируются, но наружу отдаём 200.

**Ответ при неверном secret:** `401` `{ "ok": false, "error": "unauthorized" }`

### Endpoint: `GET /api/health`
**Ответ 200:** `{ "status": "ok", "ts": "2026-07-08T14:32:00Z" }`

### Схема callback_data (payload inline-кнопок)
Формат: `action:arg1:arg2`. Лимит Telegram — 64 байта, поэтому используем короткие
slug'и, не UUID.

| callback_data | Действие |
|---------------|----------|
| `menu` | Показать меню |
| `cart` | Показать корзину |
| `about` | Раздел «О нас» |
| `item:<slug>` | Тап по позиции (если есть вариант — спросить вариант, иначе добавить) |
| `var:<slug>:<variant>` | Выбран вариант (`syrup_yes`/`syrup_no`/`tea_green`/`tea_black`) |
| `add:<slug>` | Добавить безвариантную позицию в корзину |
| `inc:<idx>` / `dec:<idx>` | Изменить количество строки корзины по индексу |
| `want:<slug>` | Тап «хочу 👀» по отсутствующей позиции |
| `checkout` | Начать оформление |
| `lm:<code>` | Выбор ориентира (`entry`/`middle`/`tail`/`text`) |
| `car_same` / `car_new` | Быстрый повтор машины / ввести новую |
| `pay:<method>` | Выбор оплаты (`sbp`/`cash`/`terminal`) |
| `adm_status:<order_uuid>:<status>` | Смена статуса заказа (admin) |
| `adm_stop:<slug>` | Переключить наличие позиции (admin, из `/стоп`) |

### Команды (текстовые сообщения)
| Команда | Контекст | Действие |
|---------|----------|----------|
| `/start` | private | Сброс корзины/state, приветствие + главное меню |
| `/init` | group | Сохранить `chat.id` как `admin_chat_id`. Ответ: «✅ Этот чат — рабочий» |
| `/стоп` | admin group | Клавиатура наличия позиций (кнопки `adm_stop:<slug>`) |
| `/стат` | admin group | Дневная сводка |
| `/menu` | private | Синоним кнопки «Меню» |
| прочий текст | private, state=checkout_car | Принять как «машина» |
| прочий текст | private, state=checkout_landmark_text | Принять как ориентир словами |
| прочий текст | private, иное | Мягкая подсказка + кнопка «Меню» |

### Внутренняя маршрутизация (псевдо)
```
if update.callback_query:
    verify secret (already at HTTP layer)
    session = getOrCreateSession(from.id, from.username)
    route by callback_data.action
    answerCallbackQuery(cb.id)   // всегда, иначе крутится «часики»
elif update.message:
    if chat.type == 'group': route admin commands (check admin_chat_id for /стоп,/стат)
    else: route private (commands, or text-by-state)
return 200 {ok:true}
```

### Исходящие вызовы Telegram Bot API (используемые методы)
- `sendMessage(chat_id, text, reply_markup, parse_mode='HTML')`
- `editMessageText(chat_id, message_id, text, reply_markup)` — для карточек статуса и корзины
- `answerCallbackQuery(callback_query_id, text?)` — гасим «часики», иногда toast
- `deleteMessage` — не используем в MVP (редактируем на месте)

**Пример: карточка заказа в admin-чат (sendMessage):**
```json
{
  "chat_id": -1001234567890,
  "parse_mode": "HTML",
  "text": "🆕 <b>Заказ #7</b> · 14:32\n• Айс-латте (с сиропом) ×2\n• Сэндвич сыр-ветчина ×1\n<b>Итого: 750 ₽</b> · 💵 наличные\n🚗 белая гранта а123вс · середина очереди\n👤 @driver_ivan",
  "reply_markup": {
    "inline_keyboard": [[
      { "text": "🚚 Взял",     "callback_data": "adm_status:<uuid>:in_progress" },
      { "text": "✅ Доставлен", "callback_data": "adm_status:<uuid>:delivered" },
      { "text": "❌ Отмена",    "callback_data": "adm_status:<uuid>:cancelled" }
    ]]
  }
}
```
Полученный `message_id` ответа сохраняется в `orders.admin_message_id`.

### Zod-схемы валидации входа
```typescript
import { z } from "zod";

export const TgFrom = z.object({
  id: z.number().int(),
  username: z.string().optional(),
});

export const TgChat = z.object({
  id: z.number().int(),
  type: z.enum(["private", "group", "supergroup", "channel"]),
});

export const TgMessage = z.object({
  message_id: z.number().int(),
  chat: TgChat,
  from: TgFrom.optional(),
  text: z.string().max(4096).optional(),
});

export const TgCallbackQuery = z.object({
  id: z.string(),
  from: TgFrom,
  message: TgMessage.optional(),
  data: z.string().max(64).optional(),
});

export const TgUpdate = z.object({
  update_id: z.number().int(),
  message: TgMessage.optional(),
  callback_query: TgCallbackQuery.optional(),
});

// callback_data парсим и валидируем «action:args»
export const CallbackData = z.string().regex(/^[a-z_]+(:[a-zA-Z0-9_]+){0,2}$/);
```

### Ошибки и коды
| Ситуация | Поведение |
|----------|-----------|
| Неверный secret | HTTP 401, тело `{ok:false,error:"unauthorized"}` |
| Невалидный Update (Zod fail) | Лог + HTTP 200 `{ok:true}` (не ретраить) |
| Ошибка БД при заказе | Лог; клиенту «⚠️ Что-то пошло не так, повторите»; state не сбрасываем |
| Telegram API 429 (flood) | Одна повторная попытка с задержкой из `retry_after`, потом лог |
| Позиция снята пока собирал корзину | При оформлении фильтруем недоступные, сообщаем клиенту |

## БЛОК 4: Диалоговые сценарии (UI бота)

> Аналог UI/UX-экранов. «Экран» бота = сообщение с текстом и inline-клавиатурой.
> Для каждого — точный текст, клавиатура, состояние FSM, действия и переходы.
> Все состояния хранятся в `user_sessions.state`. Три «сервисных» состояния
> (аналог Loading/Empty/Error) описаны отдельно в конце блока.

### FSM: карта состояний
```
browsing ──item(с вариантом)──> choosing_variant ──var──> cart
browsing ──add(без варианта)──────────────────────────> cart
cart ──checkout──> checkout_landmark ──lm:text──> checkout_landmark_text ──┐
                              │ lm:entry/middle/tail                        │
                              ▼                                             ▼
                       (если last_car есть) car_same/car_new         checkout_car
                              │                                             │
                              ▼                                             ▼
                       checkout_payment <───────────────────────────── checkout_car
                              │ pay:*
                              ▼
                       (заказ создан) → browsing (корзина очищена)
```

### Экран: Приветствие (/start)
**State:** browsing (сброс cart и context)
**Текст:**
```
🐱 Bubble Cat приехал к очереди!

Напитки и сэндвичи прямо к вашей машине — не выходя из очереди.
Соберите заказ, укажите машину, курьер принесёт. Оплата при получении.
```
**Клавиатура:**
```
[ 🧋 Меню ]
[ 🛒 Мой заказ ]  [ ℹ️ О нас ]
```
**Действия:** `menu` → Меню; `cart` → Корзина; `about` → О нас.

### Экран: Меню
**State:** browsing
**Текст (шапка):** `🧋 Что берём? Всё по 500 мл.`
**Клавиатура (динамическая из БД):**
- Секция «в наличии» (`field_item=true AND available=true`), по кнопке на позицию:
  `[ Кислый лимонад — 250₽ ]` … `callback_data = item:<slug>`
- Разделитель-заголовок текстом во втором сообщении или в той же клавиатуре кнопкой-пустышкой:
  `👀 Сегодня нет — жми, если хотел бы:`
- Секция «сегодня нет» (`available=false`): `[ <title> 👀 ]` → `want:<slug>`
- Внизу: `[ 🛒 Мой заказ (N) ]` — N = число позиций в корзине
**Действия:**
- `item:<slug>` → если `variant_group` не NULL → Экран выбора варианта (state=choosing_variant),
  иначе добавить в корзину и показать Корзину
- `want:<slug>` → `bump_demand`, ответ toast + сообщение со ссылкой на группу
**Responsive:** N/A (Telegram сам адаптирует). Кнопки — по 1–2 в ряд для читабельности
на телефоне.

### Экран: Выбор варианта
**State:** choosing_variant (в context: `{ "slug": "ice_latte" }`)
**Текст (латте):** `Айс-латте — с сиропом или без? Сироп бесплатно 🙂`
**Клавиатура:** `[ Без сиропа ]  [ С сиропом ]` → `var:ice_latte:syrup_no` / `var:ice_latte:syrup_yes`
**Текст (чай):** `Холодный чай — какой?`
**Клавиатура:** `[ Зелёный ]  [ Чёрный ]` → `var:cold_tea:tea_green` / `var:cold_tea:tea_black`
**Действия:** `var:*` → добавить строку в cart с человекочитаемым `variant`
(«С сиропом»/«Без сиропа»/«Зелёный»/«Чёрный») → Корзина.

### Экран: Корзина
**State:** cart
**Текст:**
```
🛒 Ваш заказ:
1. Айс-латте (с сиропом) ×2 — 500₽
2. Сэндвич сыр-ветчина ×1 — 250₽
Итого: 750₽
```
**Клавиатура:** на каждую строку ряд `[ − ] [ Айс-латте ×2 ] [ + ]`
(`dec:<idx>` / `inc:<idx>`), затем:
```
[ ✅ Оформить ]
[ ➕ Добавить ещё ]
```
**Действия:** `inc/dec` → пересчёт (dec при qty=1 удаляет строку); `checkout` →
если корзина пуста, toast «Корзина пуста»; иначе Экран ориентира; `menu` → Меню.

### Экран: Оформление — Ориентир
**State:** checkout_landmark
**Текст:** `Шаг 1/3. Где вы стоите в очереди?`
**Клавиатура:**
```
[ 🚩 У въезда ]  [ 🚗 Середина ]
[ 🐢 Хвост ]     [ 📍 Напишу словами ]
```
**Действия:** `lm:entry|middle|tail` → сохранить ориентир, перейти к машине;
`lm:text` → state=checkout_landmark_text, текст «Напишите ориентир одним сообщением».

### Экран: Оформление — Машина
**State:** checkout_car (или checkout_landmark_text → затем сюда)
**Текст (нет last_car):** `Шаг 2/3. Опишите машину одним сообщением: цвет, марку, номер.\nНапример: белая Гранта А123ВС`
**Текст (есть last_car):** `Шаг 2/3. Машина та же — <last_car>?`
**Клавиатура (если last_car):** `[ ✅ Да, та же ]  [ ✏️ Другая ]` → `car_same` / `car_new`
**Действия:** текстовое сообщение в state=checkout_car → сохранить как car, перейти
к оплате; `car_same` → взять last_car → оплата; `car_new` → просьба ввести текст.

### Экран: Оформление — Оплата
**State:** checkout_payment
**Текст:** `Шаг 3/3. Как оплатите курьеру при получении?`
**Клавиатура:** `[ 📱 СБП ]  [ 💵 Наличные ]  [ 💳 Терминал ]` → `pay:sbp|cash|terminal`
**Действия:** `pay:*` → создать заказ (next_order_number, insert orders, снапшот
корзины), отправить карточку в admin-чат, сохранить admin_message_id, очистить
корзину, state=browsing → Экран подтверждения.

### Экран: Подтверждение клиенту
**Текст:**
```
✅ Заказ #7 принят!
• Айс-латте (с сиропом) ×2
• Сэндвич сыр-ветчина ×1
Итого: 750₽ · 💵 наличные
🚗 белая гранта а123вс · середина очереди

Курьер подойдёт в течение 15–30 минут 🐾
Пока ждёте — загляните в нашу группу 🐱
```
**Клавиатура:** `[ 🐱 Наша группа ]` (url) `[ 🧋 Заказать ещё ]` (menu)

### Экран: О нас
**Текст:** `Bubble Cat — бабл-ти и сэндвичи 🐱\nМы на карте и в Telegram:`
**Клавиатура:** `[ 📍 На карте ]`(url) `[ 🐱 Telegram-группа ]`(url) `[ ⬅️ Меню ]`(menu)
URL-адреса берутся из env: `NEXT_PUBLIC_MAP_URL`, `NEXT_PUBLIC_GROUP_URL`.

### Admin-экран: Карточка заказа
См. Блок 3 (пример sendMessage). Текст + 3 кнопки статуса. При смене статуса
`editMessageText`: в шапке меняется значок (🆕→🚚→✅/❌), добавляется строка
`⏱ взял @courier 14:35` / `✅ доставлен 14:48`; после delivered/cancelled клавиатура
пустая.

### Admin-экран: /стоп (наличие)
**Текст:** `Наличие позиций (тап — переключить):`
**Клавиатура:** по кнопке на полевую позицию: `[ Айс-латте ✅ ]` / `[ Кофе-тоник ⛔ ]`
→ `adm_stop:<slug>`. После тапа — `editMessageText` с обновлённым значком.

### Admin-экран: /стат
**Текст (пример):**
```
📊 Сегодня (8 июля):
Заказов: 12 · Выручка: 3 400₽
💵 нал: 1 900₽ · 📱 СБП: 1 000₽ · 💳 терминал: 500₽

🔥 Заказывали:
Айс-латте — 7 · Кислый лимонад — 5 · Сэндвич курочка — 4

👀 Хотели (нет в наличии):
Ядерный зелёный — 14 · Кислые ягоды — 9
```

### Три сервисных состояния (аналог Loading / Empty / Error)
- **«Loading»:** каждое действие сразу подтверждается `answerCallbackQuery` (гасит
  «часики»); тяжёлых операций нет, спиннеры не нужны.
- **«Empty»:** корзина пуста → на `checkout`/`cart` toast «Корзина пуста, загляните в
  меню» + кнопка «Меню». Меню без доступных позиций → текст «Сегодня всё разобрали 🙈,
  но вот что хотят другие» + секция 👀.
- **«Error»:** ошибка БД/Telegram → клиенту «⚠️ Ой, что-то сломалось. Повторите,
  пожалуйста» без сброса состояния; в admin-чат опционально лог-строка.

## БЛОК 5: Business Logic

### Правила валидации
| Поле | Правило | При нарушении |
|------|---------|---------------|
| Количество позиции | integer 1–20 | `[+]` выше 20 — toast «Больше 20 не принимаем 🙂»; `[−]` ниже 1 — удалить строку |
| Ориентир (кнопка) | один из entry/middle/tail | невозможен иной ввод (кнопки) |
| Ориентир (текст) | 1–100 символов, не только пробелы | пустой → повтор «Напишите ориентир» |
| Машина | текст 3–100 символов | <3 симв. → «Слишком коротко, опишите машину подробнее» |
| Оплата | один из sbp/cash/terminal | невозможен иной ввод (кнопки) |
| callback_data | regex `^[a-z_]+(:[a-zA-Z0-9_]+){0,2}$` | не матчит → игнор + answerCallbackQuery |
| Позиция при добавлении | `available=true` | недоступна → toast «Уже разобрали 🙈» |

### Бизнес-правила
1. **Корзина живёт в сессии, не бессрочно.** При `/start` корзина обнуляется.
   Сессии старше 24 часов можно чистить (не критично для MVP; см. cron ниже —
   опционально).
2. **Заказ — это снапшот.** В `orders.items` пишется копия корзины (title, variant,
   price_kop, qty) на момент оформления. Последующие изменения меню на заказ не влияют.
3. **Номер заказа сквозной и атомарный** через `next_order_number()` — гонки
   serverless-инстансов исключены.
4. **admin_chat_id — singleton.** Последний `/init` перезаписывает. Если не задан —
   заказы сохраняются, карточки не уходят, в лог пишется предупреждение.
5. **Наличие — источник истины `menu_items.available`.** Меню всегда рендерится из БД,
   никакого кэша в памяти (serverless всё равно не сохранит).
6. **Спрос считаем без дедупликации** по пользователю (в MVP каждый тап = сигнал
   интереса; накрутка не критична для оценочного решения «что привезти»).
7. **Часовой пояс всех отчётов — Europe/Moscow.** «Сегодня» в `/стат` = сутки по МСК.
8. **Отменённые заказы** не входят в выручку `/стат`, но остаются в таблице.

### Определение admin против client
- Приватный чат (`chat.type=="private"`) → всегда client-поток.
- Сообщение/callback из группы, где `chat.id == admin_chat_id` → admin-поток.
- `/init` — единственная команда, работающая из ЛЮБОЙ группы (иначе первую привязку
  не сделать). После привязки прочие admin-команды проверяют совпадение chat.id.
- Спуфинг: callback_data со статусами (`adm_status`) обрабатываются, только если
  апдейт пришёл из admin-чата. Клиент физически не может прислать такой callback
  (кнопок нет в его чате), но проверку источника всё равно делаем.

### Аутентификация
Классической аутентификации нет: пользователь идентифицируется Telegram `from.id`,
доверяем Telegram. Admin-доступ = физическое присутствие в рабочем чате + `/init`.
Секрет вебхука (`TELEGRAM_WEBHOOK_SECRET`) защищает endpoint от постороннего POST.

### Внешние интеграции
**Telegram Bot API** (единственная):
- Отправка: sendMessage, editMessageText, answerCallbackQuery.
- Retry: при 429 читаем `parameters.retry_after`, ждём, одна повторная попытка.
  При 5xx — одна повторная через 500 мс. Дальше — лог, апдешефолт (клиент увидит
  «повторите»).
- Fallback при недоступности Telegram: сам вебхук не вызвать, если Telegram лежит,
  поэтому обрабатывать нечего. Для исходящих карточек в admin — если не ушла,
  заказ уже в БД, владелец увидит его в Supabase Table Editor (страховка).

### Безопасность
- **Секрет вебхука:** проверка заголовка `X-Telegram-Bot-Api-Secret-Token`.
- **Service-role ключ** только в серверном коде (env), никогда не в клиент.
- **Input sanitization:** пользовательский текст (машина, ориентир) вставляется в
  HTML-сообщения → экранировать `<`, `>`, `&` перед `parse_mode=HTML` (или слать
  без parse_mode для пользовательских полей). Обязательно: экранирование `car` и
  текстового `landmark` в карточке admin.
- **SQL-инъекции:** исключены — только параметризованные запросы Supabase JS /
  RPC, без конкатенации строк.
- **Rate limiting:** Telegram сам ограничивает частоту на пользователя; на нашей

- **CORS:** не применимо (нет браузерного клиента; endpoint дёргает только Telegram).

## БЛОК 6: Edge Cases

### Сеть и Telegram
1. **Telegram ретраит апдейт** (не получил 200 вовремя). Триггер: наша функция
   упала/зависла. Поведение: всегда возвращаем 200; обработка идемпотентна там, где
   можно (см. #2). Дубликат апдейта не должен создавать второй заказ — см. #12.
2. **answerCallbackQuery не отправлен** → у клиента крутятся «часики». Поведение:
   вызывать answerCallbackQuery в самом конце обработки ЛЮБОГО callback, даже при
   ошибке (в finally).
3. **Медленный Supabase / таймаут функции Vercel.** Триггер: пик нагрузки. Поведение:
   операции короткие; при таймауте клиент повторяет действие, состояние в БД не
   повреждено (пишем атомарно, заказ — одной вставкой).
4. **Telegram 429 при отправке карточки в admin.** Поведение: одна повторная попытка
   по `retry_after`; заказ уже в БД, не теряется.

### Данные
5. **Позицию сняли (`/стоп`), пока клиент собирал корзину.** Триггер: гонка. При
   `checkout` перепроверяем `available` каждой строки; недоступные убираем, клиенту:
   «Айс-латте закончился, убрали из заказа. Оформить остальное?».
6. **Пустая корзина на оформлении.** См. Empty-состояние: toast + «Меню».
7. **Клиент жмёт старую кнопку из прошлого сообщения** (например, `inc:3`, а строки 3
   уже нет). Поведение: проверяем границы индекса; если нет — toast «Корзина
   изменилась» + перерисовать корзину.
8. **Очень длинный текст машины/ориентира** (>100 симв). Обрезаем до 100 + сообщаем.
9. **Пользователь меняет @username между заказами.** Пишем актуальный на момент
   апдейта; в старых заказах остаётся прежний (снапшот). Норм.
10. **Юзер без @username** (скрыт). `username=NULL`; в карточке показываем `id` вместо
    `@—`. Курьер ориентируется по машине, не по нику.
11. **Параллельные заказы двух клиентов** в один момент. `next_order_number()`
    атомарна — номера не совпадут.

### Безопасность / злоупотребления
12. **Двойной тап «Оформить»/`pay:*`** (дабл-клик или ретрай апдейта) → риск двух
    заказов. Защита: при входе в создание заказа проверяем `state==checkout_payment`;
    сразу переводим state в `browsing` ДО отправки в Telegram; повторный `pay:*`
    придёт уже в state=browsing и будет проигнорирован (toast «Заказ уже оформлен»).
13. **Подмена callback `adm_status`** клиентом. Обрабатываем только из admin-чата;
    из привата — игнор.
14. **Спам «хочу 👀»** одним человеком ради накрутки. Принимаем (см. правило 6);
    при желании владелец интерпретирует числа как порядок величины, не точные.
15. **Посторонний POST на `/api/telegram`.** Без верного secret — 401, ничего не
    обрабатывается.
16. **XSS через поле «машина»** (клиент пишет `<b>x</b>`). Экранируем перед HTML
    (утилита `esc`), тег не исполнится.
17. **Кто-то чужой пишет `/init` в другой группе.** Перезапишет admin_chat_id. Риск
    низкий (нужно знать бота и добавить его в группу). Митигация MVP: после первого
    `/init` последующие `/init` требуют, чтобы отправитель был тем же, кто привязал
    (сохраняем `admin_bound_by`), иначе — «Чат уже привязан».

### Лимиты
18. **callback_data > 64 байт.** Исключено дизайном (короткие slug); длинные UUID в
    callback не кладём (кроме `adm_status:<uuid>` — UUID 36 симв + префикс < 64, ок).
19. **Клавиатура меню разрастётся**, когда владелец добавит много позиций «сегодня
    нет». Telegram лимит ~100 кнопок/сообщение. Митигация: секцию 👀 в MVP держать
    ≤ 30 позиций; при росте — отдельное сообщение «показать ещё».
20. **Много заказов в /стат** (сотни). Запрос агрегирующий по дате с индексом
    `idx_orders_created` — быстрый. JSONB-агрегация позиций — по разумным объёмам ок.

### Время
21. **«Сегодня» на границе полуночи МСК.** `/стат` фильтрует по МСК-суткам
    (`created_at AT TIME ZONE 'Europe/Moscow'`), не по UTC — иначе вечерние заказы
    улетят во «вчера».
22. **Сессия висит с прошлого дня** (клиент бросил на середине checkout). При новом
    `/start` — сброс. При старом callback в старом state — обрабатывается штатно,
    данные в context ещё валидны (меню могло смениться → перепроверка available на
    checkout, #5).

## Приложение А: Чеклист ручного запуска (для владельца, ~20 минут)

> Эти шаги Claude Code выполнить не может — они требуют аккаунтов. Claude Code
> сгенерирует этот же чеклист в README с подставленными значениями.

1. **BotFather:** `/newbot` → имя, username → скопировать токен.
2. **Supabase:** создать проект → SQL Editor → выполнить весь SQL из Блока 2
   (таблицы, сид, RPC, триггеры) одним прогоном.
3. **Скопировать** `Project URL` и `service_role` ключ (Settings → API).
4. **Vercel:** импортировать репозиторий → задать env-переменные (Блок 5) → Deploy.
5. **Установить webhook** (одна команда, curl, с secret):
   `curl "https://api.telegram.org/bot<TOKEN>/setWebhook?url=https://<app>.vercel.app/api/telegram&secret_token=<SECRET>"`
6. **Рабочий чат:** создать группу, добавить бота, отправить `/init`.
7. **Проверить:** написать боту `/start` в личке → пройти заказ → карточка пришла в
   группу. Отметить статусы. `/стат` в группе.
8. **QR:** сгенерировать QR на `t.me/<bot_username>` → на флаер.
