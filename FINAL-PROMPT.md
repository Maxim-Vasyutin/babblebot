Ты собираешь production-ready Telegram-бот приёма заказов **автономно**, пока меня
нет за компьютером. Полная спецификация — в `SPEC.md`, правила проекта — в
`CLAUDE.md` и `.claude/rules/`, специализированные субагенты — в `.claude/agents/`.
Прочитай их ПЕРЕД началом.

## Суть продукта (чтобы ты понимал, что строишь)
Бот принимает заказы напитков и сэндвичей от водителей, стоящих в очередях на АЗС.
Клиент в личке бота собирает корзину, указывает машину и место в очереди → заказ
карточкой прилетает в рабочий Telegram-чат заведения → курьер доставляет к машине,
оплата при получении. Дополнительно бот копит статистику спроса на позиции, которых
сегодня нет (кнопки «хочу 👀»). Это MVP на 1–2 дня акции, цель — провалидировать
гипотезу, не построить масштабную платформу.

## Три архитектурных факта, без которых всё сломается
1. **Serverless. Состояние — в БД, не в памяти.** Один webhook-запрос = одна жизнь
   функции. FSM диалога и корзина живут в таблице `user_sessions`. Никаких Map или
   глобальных переменных для сессий — они не переживут следующий вызов.
2. **Доступ к БД только service-role ключом с сервера.** Нет Supabase Auth, нет
   публичного клиента. RLS включена как deny-all (defense-in-depth), НЕ как контур
   авторизации. Не пиши политик `auth.uid() = user_id` — auth.users в проекте нет,
   `orders.user_id` это Telegram ID.
3. **Webhook всегда отвечает 200** (кроме неверного secret → 401), иначе Telegram
   ретраит апдейт и плодит дубликаты.

## Стек (не отклоняйся)
Next.js 16 (App Router), TypeScript strict, Supabase (Postgres, service-role), Zod,
нативный fetch к Telegram Bot API. Деплой — Vercel. НЕТ фронтенда, React-страниц,
shadcn, Tailwind, Supabase Auth.

## Порядок работы (делегируй субагентам)
1. **database-architect** — перенеси схему из SPEC.md Блок 2 в `supabase/migrations/`
   точно как есть: 5 таблиц, 2 RPC (`next_order_number`, `bump_demand`), сид меню
   (6 позиций), moddatetime-триггеры, deny-all RLS. Плюс `.env.example` со всеми
   переменными из Блока 5.
2. **backend-engineer** — webhook route `/api/telegram` (проверка secret, Zod-валидация
   Update, service-role клиент, контракт «всегда 200», retry для 429/5xx),
   `/api/health`, утилиты Bot API и `esc()`.
3. **telegram-bot-engineer** — основной объём: FSM по карте из SPEC.md Блок 4
   (browsing → choosing_variant → cart → checkout_landmark → checkout_car →
   checkout_payment), клиентский и admin потоки, callback-протокол из Блока 3, тексты
   сообщений и клавиатур дословно как в Блоке 4. Варианты латте (сироп/без) и чая
   (зелёный/чёрный) спрашивать ДО корзины. Заложить защиту от двойного заказа (Блок 6

4. **qa-tester** — ОДИН проход в конце (не после каждого модуля, чтобы не жечь

## Граница автономности — критично
Ты НЕ выполняешь шаги, требующие реальных аккаунтов/токенов. Конкретно НЕ делай:
- создание Supabase-проекта, выполнение SQL в дашборде
- Vercel deploy, задание env в Vercel
- установку webhook (`curl .../setWebhook`) — только ЗАПИШИ команду в README с
  плейсхолдерами `<TOKEN>`, `<SECRET>`, `<APP_URL>`, НЕ выполняй
- `/init` в группе, любые сетевые вызовы с реальными секретами

Вместо этого сгенерируй `README.md` с чеклистом ручного запуска (Приложение А
SPEC.md) — по шагам, с плейсхолдерами. Код должен собираться (`npm run build`) и
проходить lint с заглушками, без реальных секретов.

## Если упрёшься в шаг, требующий меня
